# 代码审查技能

你已获得代码审查能力，可以对项目代码进行系统性质量审查。

## 审查维度

### 1. 代码质量
- 命名规范：变量/函数/类命名是否语义清晰、风格一致
- 函数设计：单一职责、参数数量、返回值一致性
- 代码复杂度：圈复杂度、嵌套深度、函数长度
- 重复代码：DRY 原则违反、可提取的公共逻辑

### 2. 类型安全
- TypeScript：any 类型滥用、类型断言过度、泛型约束缺失
- 运行时类型：输入校验、边界检查、null/undefined 处理
- 接口一致性：参数类型与实际使用是否匹配

### 3. 错误处理
- 异常捕获：try-catch 覆盖、catch 块是否吞掉错误
- 错误传播：Promise 链未处理 rejection、async 函数未 await
- 降级策略：失败后是否有合理回退

### 4. 性能隐患
- 内存泄漏：事件监听未移除、定时器未清理、闭包引用
- 不必要渲染：React/Vue 组件重复渲染、大列表未虚拟化
- 资源浪费：重复计算、未缓存的结果、同步阻塞操作

### 5. 安全问题
- 输入注入：XSS、命令注入、SQL 注入风险
- 敏感数据：硬编码密钥、日志泄露、明文存储
- 权限控制：越权访问、未验证的用户输入

### 6. 架构设计
- 依赖关系：循环依赖、紧耦合、抽象层次不一致
- 模块边界：职责划分、接口设计、扩展性
- 状态管理：状态提升、数据流方向、副作用隔离

## 审查工作流

1. **确定范围**：明确审查的文件/模块范围
2. **静态扫描**：运行 linter 和 typecheck（execute_command）
3. **逐文件审查**：读取源码，按维度逐项检查
4. **交叉引用**：通过 content_search 追踪调用链和依赖
5. **生成报告**：按严重程度分级（🔴 严重 / 🟡 警告 / 🔵 建议）

## 报告格式

```
## 代码审查报告

### 概览
- 审查范围：[文件列表]
- 发现问题：🔴 X 严重 / 🟡 Y 警告 / 🔵 Z 建议

### 🔴 严重问题
1. [文件:行号] 问题描述 → 修复建议

### 🟡 警告
1. [文件:行号] 问题描述 → 改进建议

### 🔵 优化建议
1. [文件:行号] 描述 → 建议方案

### 总结与优先级
1. 最需优先处理的问题
2. 整体代码质量评价
3. 改进路线图
```

## 使用原则

1. **先运行工具再审查**：先执行 linter/typecheck 收集机器可检测的问题，再进行人工审查
2. **给出可操作建议**：每个问题必须附带具体的修复方案，不只是指出问题
3. **区分严重程度**：安全漏洞和逻辑错误为严重，风格问题为建议
4. **关注上下文**：审查代码时理解业务逻辑，避免脱离语境的建议
5. **渐进式审查**：大项目分模块审查，每次聚焦一个维度
