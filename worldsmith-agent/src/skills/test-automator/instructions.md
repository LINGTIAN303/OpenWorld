# 测试自动化技能

你已获得测试自动化能力，可以生成测试用例、运行测试并分析覆盖率。

## 测试类型

### 1. 单元测试
- 函数/方法级别测试
- 输入输出边界测试
- 异常路径测试
- Mock 依赖隔离

### 2. 集成测试
- 模块间交互测试
- API 端点测试
- 数据库/存储操作测试
- 事件流测试

### 3. 端到端测试
- 用户流程测试
- 组件渲染测试
- 跨模块协作测试

## 测试框架检测

优先检测项目中已有的测试框架：

| 语言 | 框架 | 检测标志 |
|------|------|----------|
| TypeScript | Vitest | vitest.config.* |
| TypeScript | Jest | jest.config.* |
| TypeScript | Mocha | .mocharc.* |
| Python | pytest | pytest.ini / conftest.py |
| Rust | cargo test | Cargo.toml |
| Go | go test | *_test.go |

如果项目没有测试框架，推荐 Vitest（TypeScript）或 pytest（Python）。

## 工作流

### TDD 模式（推荐）
1. **理解需求**：读取源码，理解函数/模块的预期行为
2. **编写测试**：先写测试用例，覆盖正常路径和边界条件
3. **运行测试**：确认测试失败（红灯）
4. **实现代码**：编写最小实现使测试通过
5. **运行测试**：确认测试通过（绿灯）
6. **重构**：优化代码，保持测试通过

### 补充测试模式
1. **扫描源码**：识别未测试的函数/模块
2. **分析风险**：按复杂度和重要性排序
3. **生成测试**：为高风险模块生成测试用例
4. **运行验证**：执行测试，确保全部通过
5. **覆盖率分析**：运行覆盖率工具，识别盲区

## 测试用例设计原则

1. **AAA 模式**：Arrange（准备）→ Act（执行）→ Assert（断言）
2. **单一断言**：每个测试只验证一个行为
3. **命名规范**：`should [expected] when [condition]`
4. **边界测试**：空值、极值、非法输入
5. **独立性**：测试间无依赖，可任意顺序执行
6. **可重复**：相同输入始终产生相同结果

## 覆盖率目标

| 类型 | 目标 | 说明 |
|------|------|------|
| 语句覆盖 | ≥ 80% | 核心模块 ≥ 90% |
| 分支覆盖 | ≥ 70% | 关键分支 100% |
| 函数覆盖 | ≥ 85% | 公共 API 100% |

## 报告格式

```
## 测试报告

### 概览
- 测试框架：[框架名]
- 测试文件：X 个
- 测试用例：X 个（✅ 通过 / ❌ 失败 / ⏭️ 跳过）
- 覆盖率：语句 X% | 分支 X% | 函数 X%

### 失败用例
1. [测试名] → 失败原因 → 修复建议

### 覆盖率盲区
1. [文件:行号] 未覆盖的分支 → 建议补充的测试

### 建议的下一步
1. 优先补充的测试
2. 需要重构以提升可测试性的代码
```

## 使用原则

1. **优先检测已有框架**：不要引入新框架，除非项目完全没有测试
2. **测试代码质量等同生产代码**：测试也要清晰、可维护
3. **避免过度 Mock**：只 Mock 外部依赖，内部逻辑用真实调用
4. **运行测试验证**：生成的测试必须实际运行并确认结果
5. **渐进式补充**：从最关键的模块开始，逐步提升覆盖率
