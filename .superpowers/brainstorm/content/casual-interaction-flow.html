2. 发言欲望计算细节

欲望值的四个维度及其权重

欲望值 = f(基础, @加成, 角色, 活跃度)
🧮 计算公式
desire = base(0.3) + mentionBoost(×@时+0.5) + roleRelevance(msg关键词匹配) × weight(0.3) - recentDecay(近N条发言数 × 0.1) - muted? → 0 最终值 clamp [0, 1] 阈值默认 0.4 → 超过才发言
各维度说明：
维度 范围 作用
基础欲望固定 0.3保证每个Agent都有一定概率参与
@提及加成+0.5被@时基本必回（0.3+0.5=0.8）
角色相关度0 ~ 0.3通过角色描述/关键词匹配计算
活跃衰减-0.1 × 近期条数防止单个Agent连续刷屏
禁言归零= 0被禁言时无论其他维度多高都不发言

关键特性：
  • idle ↔ calculating ↔ responding 循环，不会自动终止
  • 任何状态 → paused 用户可随时暂停（如费用警告）
  • responding 中新消息到来：排队等待当前批次完成后立即处理
  • 无 completed/terminated：闲聊群持续存在直到手动解散

4. 并行响应的边界情况

需要处理的异常场景
场景A：部分 Agent API 失败
成功的正常显示消息，失败的显示「XXX 回复失败」提示，不影响其他 Agent 的结果展示
场景B：所有 Agent 都失败
显示一条系统提示：「本次请求未能获得任何回复」，保留用户消息在聊天记录中
场景C：用户快速连续发送多条消息
第二条消息排队等待，第一条的响应全部完成后才开始第二批的计算和调用
场景D：费用超预算
暂停引擎，显示费用超支提示，用户确认后再恢复或终止
场景E：上下文窗口溢出
复用现有 ContextManager 的摘要策略，自动压缩历史消息
