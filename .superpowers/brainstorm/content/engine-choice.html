渲染引擎选择

这是整个重构的地基决策，影响所有后续设计

当前状态

现有思维导图使用 Cytoscape.js 渲染，它是一个成熟的图可视化库，但在自定义交互（嵌套图、分组框、涂鸦层）方面存在局限。

A

继续 Cytoscape.js — 增量扩展

在现有 Cytoscape 基础上扩展，利用其复合节点(compound nodes)实现分组框，用事件系统实现嵌套导航。

B

Canvas 2D 自研引擎 — Project Graph 同款路线

完全自研 Canvas 2D 渲染层，像 Graphif 一样获得完全控制权，但开发量最大。

C

混合方案 — Cytoscape 核心 + Canvas 叠加层

保留 Cytoscape 处理图布局和节点渲染，在其上叠加 Canvas 层处理涂鸦/批注，用 Vue 组件实现分组框和详情侧栏。

方案 A: 继续Cytoscape

  • 改动最小，现有代码可复用
  • 复合节点原生支持分组框
  • 布局算法成熟
  • 事件系统完善

方案 A 的局限

  • 嵌套图需自行实现（Cytoscape 无原生支持）
  • 涂鸦层无法在 Cytoscape 内实现
  • 自定义渲染受限于 Cytoscape API
  • 性能受 SVG/Canvas 切换影响

方案 B: Canvas 自研

  • 完全控制渲染和交互
  • 性能最优（Graphif 验证过）
  • 嵌套图/涂鸦/分组框无限制
  • 可实现任意视觉效果

方案 B 的代价

  • 开发量极大（需自研布局算法）
  • 需重写所有现有功能
  • 文本渲染/交互需手动处理
  • 无障碍访问难以实现

方案 C: 混合方案 (推荐)

  • 保留 Cytoscape 的布局和图算法优势
  • Canvas 叠加层实现涂鸦/批注
  • Vue 组件实现复杂 UI（详情栏/分组框）
  • 渐进式迁移，风险可控
  • 嵌套图用路由+状态管理实现

方案 C 的挑战

  • 多层渲染需要坐标同步
  • Cytoscape 与外部组件的交互桥接
  • 分组框需要复合节点+CSS叠加
