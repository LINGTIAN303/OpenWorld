⚠️ 风险分析与对策

多模型计费复杂 已解决

CostTracker 已按 agentId 独立追踪，calculateCost() 已支持不同模型不同币种。 只需在 recordUsage 时传入 modelId，用对应价格计算即可。

不同 API 密钥 需处理

用户可能只配置了一个 provider 的 API key。选择其他 provider 时需要提示"未配置 API Key"。 方案：模型列表只显示已配置 API key 的 provider 的模型，未配置的灰显并提示。

并行 vs 异步 无需改动

当前设计是顺序发言（轮次制），每个 agent 独立 await。多模型不影响执行模式。 未来如需并行发言（多个 agent 同时思考），只需改为 Promise.allSettled()，与模型选择无关。

单模型失败风险 需增强

当前 catch 块已处理单个 agent 失败（显示"发言失败"系统消息），但缺少重试和降级。 方案：失败后自动跳过该 agent 继续，3次连续失败则提示用户是否终止。

上下文窗口差异 需处理

不同模型上下文窗口差异巨大（8K vs 1M）。ContextManager 需要按当前发言者的模型动态调整策略。 当前使用固定 contextWindow=32768，需改为从 participant.providerConfig 读取。

🎯 推荐方案：渐进式多模型支持

Phase 1（本轮实现）：
  • 配置页：每个参与者添加模型选择下拉框
  • 模型列表：只显示已配置 API key 的 provider 的模型
  • 默认模型：使用全局设置的当前模型
  • CostTracker：recordUsage 传入 modelId，用 calculateCost() 精确计费
  • ContextManager：按发言者模型动态调整上下文窗口
  • 错误增强：连续失败自动跳过 + 提示
Phase 2（未来可选）：
  • 并行发言模式（多 agent 同时思考）
  • 模型健康检测（启动前验证 API 可达性）
  • 自动降级（主模型失败时切换备用模型）
